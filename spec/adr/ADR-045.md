# ADR-045 — RerankerProtocol: Post-Retrieval Reranking

**Status:** Accepted  
**Phase:** 10 (tutorial prerequisite)  
**Amends:** ADR-040 (RagService API)

## Context

ADR-040 deferred reranking: "reranking is valuable but requires a cross-encoder model or API call. Can be added as an optional post-retrieval step in a future phase without breaking the current API."

Production RAG pipelines universally include a reranking step between retrieval and generation. Without it, the LLM receives noisy vector search results (approximate nearest neighbors), leading to lower quality answers. The tutorial (Phase 10) builds an agentic Document Q&A Assistant that should demonstrate best-practice RAG.

## Decision

### RerankerProtocol

```python
from dataclasses import dataclass
from typing import Protocol


@dataclass(frozen=True)
class RerankResult:
    """A reranked document with its relevance score."""
    content: str
    score: float
    index: int  # original position in the input list


class RerankerProtocol(Protocol):
    async def rerank(
        self,
        query: str,
        documents: list[str],
        *,
        top_k: int = 5,
    ) -> list[RerankResult]:
        """Rerank documents by relevance to the query.

        Args:
            query: The user's question
            documents: List of document texts to rerank
            top_k: Number of top results to return

        Returns:
            Top-k documents ordered by relevance score (highest first)
        """
        ...
```

### Built-in Backends

| Backend | Extra | Implementation |
|---------|-------|---------------|
| Ollama cross-encoder | `reranker-ollama` | Calls Ollama's `/api/embed` or compatible rerank endpoint |
| OpenAI-compatible | `reranker-openai` | Calls any OpenAI-compatible rerank API (Jina, Cohere via proxy) |

Both backends use the same construction pattern (ADR-012): receive `Settings` at init, read `reranker_model` and `reranker_url` from settings.

### Settings

```python
reranker_provider: str = "none"  # "ollama" | "openai" | dotted path | "none"
reranker_model: str = "jina-reranker-v2-base-multilingual"
reranker_url: str = "http://localhost:11434"
reranker_timeout: float = 30.0  # I22 compliance
```

### RagService Amendment

`RagService` gains an optional `reranker` parameter:

```python
class RagService:
    def __init__(
        self,
        embedding: EmbeddingProtocol,
        vector_store: VectorStoreProtocol,
        *,
        reranker: RerankerProtocol | None = None,  # NEW
        chunk_size: int = 512,
        chunk_overlap: int = 64,
        chunking_strategy: str = "fixed",
    ) -> None:
        ...
```

When `reranker` is provided, `retrieve()` becomes:
1. Vector search with `top_k * 3` (over-fetch)
2. Rerank results against the query
3. Return top `top_k` reranked results

When `reranker` is None, behavior is unchanged (direct vector search results).

## Consequences

- ADR-040 `RagService` constructor gains optional `reranker` kwarg (backwards compatible)
- New extras: `reranker-ollama` (httpx for Ollama API), `reranker-openai` (openai or httpx)
- I1 applies: both backends must fully implement `RerankerProtocol`
- I22 applies: `reranker_timeout` setting enforced
- I3 applies: extras-gated imports
- ARCHITECTURE.md Protocol methods table gains a Reranker row
- GLOSSARY.md gains `RerankerProtocol`, `RerankResult`
- Tutorial Part 5 uses the reranker to demonstrate best-practice RAG

## Rejected Alternatives

- **Reranking inside VectorStoreProtocol** — rejected: reranking is query-time logic, not a store responsibility. Separation of concerns.
- **Always rerank (no opt-out)** — rejected: reranking adds latency. Simple use cases (small doc sets, low-latency requirements) should skip it.
- **Cohere/Jina as named backends** — rejected: these are accessible via the OpenAI-compatible endpoint. No need for per-vendor backends when the API shape is the same.
