# ADR-044 — Alembic Branches for Framework Migrations

**Status:** Accepted  
**Phase:** Post-release fix (supersedes I16 migration architecture)  
**Amended by:** ADR-048 (user branch definition)

## Context

The current framework migration system (I16) uses separate Alembic "instances" per module — each with its own `env.py`, `script_location`, and independent `command.upgrade(cfg, "head")` calls in `cli/db.py`. All modules share a single `fas_alembic_version` table.

This has a correctness bug: when auth migrations run first and stamp `fas_auth_0001` in the version table, the AI migration instance sees that revision as its "current state" but can't find it in its own script directory. Alembic raises `ResolutionError`.

Additionally, the two-instance approach cannot express cross-module dependencies (e.g., AI migrations depending on auth tables existing first).

## Decision

Replace the multi-instance migration architecture with **Alembic branches** — a single Alembic instance with multiple `version_locations`, branch labels, and conditional gating via dynamic discovery in `env.py`.

### Architecture

One Alembic instance (the user's generated `alembic/`) manages:
- **User migrations** — `{project_name}` branch (amended by ADR-048), generated by `fas makemigrations`
- **Framework auth migrations** — `auth` branch, bundled in the package
- **Framework AI migrations** — `ai` branch, bundled in the package

`fas migrate` runs `alembic upgrade heads` (plural) which upgrades all active branches in dependency order.

### Dynamic Discovery (replaces hardcoded version_locations)

The generated `alembic/env.py` discovers framework migration locations at runtime:

```python
import importlib
from pathlib import Path

_FRAMEWORK_MIGRATION_GATES = [
    ("fast_agent_stack.core.auth.migrations", "pwdlib"),
    ("fast_agent_stack.core.ai.migrations", ["anthropic", "openai", "litellm", "aioboto3"]),
]

def _framework_version_locations() -> list[str]:
    locs = []
    for module_name, gate_packages in _FRAMEWORK_MIGRATION_GATES:
        gates = gate_packages if isinstance(gate_packages, list) else [gate_packages]
        if not any(_importable(pkg) for pkg in gates):
            continue
        try:
            mod = importlib.import_module(module_name)
            versions_dir = str(Path(mod.__file__).parent / "versions")
            locs.append(versions_dir)
        except ImportError:
            pass
    return locs

def _importable(name: str) -> bool:
    try:
        __import__(name)
        return True
    except ImportError:
        return False
```

This is passed to Alembic's config before running:

```python
version_locations = [
    str(Path(config.get_main_option("script_location")) / "versions"),  # user migrations
    *_framework_version_locations(),  # framework branches (conditional)
]
config.set_main_option("version_locations", " ".join(version_locations))
```

### Branch Labels

Framework migrations declare branch labels:

```python
# fast_agent_stack/core/auth/migrations/versions/0001_fas_auth_initial.py
revision: str = "fas_auth_0001"
down_revision: str | None = None
branch_labels: tuple[str, ...] = ("fas_auth",)

# fast_agent_stack/core/ai/migrations/versions/0001_fas_ai_conversation.py
revision: str = "fas_ai_0001"
down_revision: str | None = None
branch_labels: tuple[str, ...] = ("fas_ai",)
depends_on: str | tuple[str, ...] | None = "fas_auth_0001"  # AI requires auth tables
```

### Cross-Module Dependencies

AI migrations can now declare `depends_on = "fas_auth_0001"` — Alembic ensures auth tables exist before AI tables are created. This was impossible under the previous two-instance approach.

### CLI Changes

```python
# cli/db.py — simplified
@app.command()
def migrate() -> None:
    """Apply framework + user database migrations."""
    _require_alembic_dir()
    cfg = _alembic_cfg()
    # version_locations already configured in env.py dynamically
    command.upgrade(cfg, "heads")  # plural — upgrades all branches
    typer.echo("Migrations applied.")
```

The `_run_framework_migrations()` function is **deleted**. Framework migrations are now part of the same Alembic graph as user migrations.

### Generated Template Changes

`alembic/env.py.jinja`:
- Adds `_framework_version_locations()` discovery function
- Passes dynamic `version_locations` to config
- `include_object` filter still excludes framework tables from `fas makemigrations` autogeneration (unchanged from current I16)

`alembic.ini` / `alembic/` config:
- No hardcoded `version_locations` — all dynamic

### What Gets Deleted

- `fast_agent_stack/core/auth/migrations/env.py` — no longer needed
- `fast_agent_stack/core/ai/migrations/env.py` — no longer needed
- `cli/db.py: _run_framework_migrations()` — no longer needed
- `FRAMEWORK_MIGRATION_MODULES` registry in `cli/db.py` — replaced by `_FRAMEWORK_MIGRATION_GATES` in generated env.py

### Version Table

Single `alembic_version` table (Alembic default) with multiple heads — one per active branch. This replaces `fas_alembic_version`. User migrations and framework migrations coexist in the same table. This is standard Alembic multi-branch behavior.

## Consequences

- I16 is amended: framework migrations no longer have separate env.py files or a separate version table. They are branches in the user's Alembic graph.
- `fas migrate` becomes `command.upgrade(cfg, "heads")` — one call, all branches.
- `fas makemigrations` autogeneration still excludes framework tables via `include_object` filter (unchanged).
- `alembic history` shows the full graph (framework + user) — better debuggability.
- `alembic downgrade` works correctly across branches.
- Framework migration files retain `branch_labels` and optional `depends_on` declarations.
- Gate-on-importability preserved: if `pwdlib` isn't installed, auth migration versions directory isn't added to `version_locations`, so those migrations are invisible to Alembic.
- Scaffolder's `alembic/env.py.jinja` does more work (discovery function) but `cli/db.py` does less.

## Rejected Alternatives

- **Separate version tables per module** — rejected: prevents cross-module `depends_on`, makes `alembic history` incomplete, complicates downgrade. It fixes the immediate bug but introduces new limitations.
- **Single linear chain (AI down_revision points to auth)** — rejected: forces installation order. AI shouldn't require auth to be installed first. Branches with `depends_on` express the dependency without forcing a single linear history.
- **Keep current approach, just fix the table name** — rejected: leaves the cross-module dependency gap open and doesn't scale to future modules.
