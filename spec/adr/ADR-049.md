# ADR-049: Unified Secret Key and Database-Backed Admin Authentication

**Status:** Accepted
**Date:** 2026-07-13
**Deciders:** Author
**Amends:** ADR-007 (Admin: SQLAdmin)

## Context

The admin panel (`/admin`) currently authenticates via a shared `admin_secret_key` setting compared
against a form password field. This is disconnected from the user database: `fas createsuperuser`
creates a user for API auth (`/auth/token`) but that user cannot log into the admin panel.

Problems:
- Two separate auth systems for one framework
- `admin_secret_key` is a credential stored in config (bad practice)
- No permission granularity in admin (anyone with the secret has full access)
- `createsuperuser` doesn't give admin panel access, which confuses users

Django uses a single `SECRET_KEY` for all signing and authenticates the admin panel against the
user table with `is_staff` / `is_superuser` gating access.

## Decision

### 1. Remove `admin_secret_key`

The `admin_secret_key` setting is removed entirely. A single `secret_key` is used for:
- JWT token signing (API auth)
- Admin panel session cookie signing (sqladmin `AuthenticationBackend` secret)

### 2. Admin panel authenticates against the user table

`_PasswordAuth.login()` verifies email + password against the `User` model:

```python
async def login(self, request: Request) -> bool:
    form = await request.form()
    email = form.get("username", "")
    password = form.get("password", "")

    async for session in get_async_session():
        result = await session.execute(select(User).where(User.email == email))
        user = result.scalar_one_or_none()
        if user is None or not user.is_active:
            return False
        ok, _ = verify_password(password, user.password_hash or "")
        if not ok:
            return False
        if not user.is_staff and not user.is_superuser:
            return False
        request.session.update({"admin_user_id": str(user.id)})
        return True
```

### 3. Access control

- `is_staff=True` OR `is_superuser=True` -> can log into `/admin`
- `is_superuser=True` -> overrides all permission checks, sees all views
- `is_staff=True` (non-superuser) -> visibility determined by the `permissions` table
- Regular users (`is_staff=False`, `is_superuser=False`) -> rejected at login

### 4. `createsuperuser` is the bootstrap command

After `fas migrate`, the developer runs `fas createsuperuser`. This creates a user who can:
- Log into the admin panel (`/admin`)
- Authenticate via the API (`/auth/token`)
- Access all admin views (superuser permission override)

### 5. `AdminLifespanHook` signature change

```python
# Before
AdminLifespanHook(app, admin_secret_key=_settings.admin_secret_key)

# After
AdminLifespanHook(app, secret_key=_settings.secret_key)
```

The `secret_key` is used only for session cookie signing (sqladmin requirement), not as a
login credential.

### 6. Startup validation change

The startup validation rule changes from:
- Before: `admin_secret_key` must be set when admin panel is enabled
- After: `secret_key` must be set when admin panel is enabled OR JWT auth is enabled

If neither auth nor admin is enabled, `secret_key` is not required.

## Consequences

| File | Change |
|------|--------|
| `core/admin/lifespan.py` | Replace shared-secret login with database-backed user verification |
| `core/config.py` | Remove `admin_secret_key` from `BaseSettings` |
| `spec/INVARIANTS.md` | Remove `admin_secret_key` startup validation rule |
| `spec/NFR.md` | Remove `admin_secret_key` reference |
| Template `settings.py.jinja` | Remove `admin_secret_key` field |
| Template `.env.example.jinja` | Remove `ADMIN_SECRET_KEY` line |
| Template `app.py.jinja` | Change `AdminLifespanHook` constructor arg |
| `cli/new.py` | Remove `admin_secret_key` from post-scaffold notes |

### Behavioral consequences

- Single `secret_key` for all signing (JWT + admin sessions) - matches Django convention
- Admin access is user-based, not secret-based - auditable, revocable per-user
- `fas createsuperuser` is the single entry point for both API and admin access
- Permission-based view visibility enables role-based admin use
- Breaking change: existing projects must remove `admin_secret_key`, ensure `secret_key` is set,
  and run `fas createsuperuser` to regain admin access
