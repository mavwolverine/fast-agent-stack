# ADR-043 — Integration Test Strategy: Two-Tier Scaffolder Verification

**Status:** Accepted  
**Phase:** 9

## Context

The framework generates projects via Copier templates with conditional logic (Jinja2 `{% if %}` blocks, copier.yml choices). Template errors are silent — a broken conditional produces a broken project with no copier-level error. Unit tests mock everything and can't catch these. The project needs integration tests that verify generated projects actually work.

Two failure classes exist:
1. **Template syntax errors** — missing imports, wrong variable names, invalid Python after Jinja rendering. Fast to detect, no services needed.
2. **Integration errors** — migrations fail on real Postgres, Redis connectivity missing, auth flow broken end-to-end. Requires real services to detect.

## Decision

Two test tiers, run at different cadences:

### Tier 1 — Scaffold Smoke Tests (every PR)

**Trigger:** Every push / PR to main  
**Duration:** ~10–15s  
**Infrastructure:** None (no Docker, no external services)

For each preset (`minimal`, `standard`, `full`, `agent`):
1. `fastagentstack new <tmpdir> --preset <preset>` (non-interactive)
2. Verify generated files are valid Python: `py_compile` on all `.py` files
3. Verify `pyproject.toml` is parseable (tomllib)
4. Verify key files exist based on preset (e.g., `agent` preset → `agents.py` exists)
5. `pip install -e .` succeeds (deps resolve)
6. `python -c "from <project>.app import app"` succeeds (app module imports without error)

**Does NOT start a server or require a database.**

### Tier 2 — E2E Tests with Testcontainers (release gate)

**Trigger:** Before release (`v*` tag push), nightly, or manual `workflow_dispatch`  
**Duration:** ~60–90s  
**Infrastructure:** Docker (testcontainers spins up Postgres + Redis)

For each preset:
1. Scaffold project (same as Tier 1)
2. Start Postgres container (testcontainers `PostgresContainer`)
3. Start Redis/Valkey container (testcontainers `RedisContainer`)
4. Set `DATABASE_URL`, `REDIS_URL`, `SECRET_KEY` env vars pointing to containers
5. `fastagentstack migrate` — verify exit code 0
6. Start app via `uvicorn` in a subprocess
7. Hit endpoints and verify:

| Preset | Endpoints verified |
|--------|-------------------|
| `minimal` | `GET /` → 200, `GET /health/live` → 200 |
| `standard` | above + `GET /health/ready` → 200 (DB+Redis), `POST /auth/token` → 401 |
| `full` | above + response has `X-RateLimit-Limit` header |
| `agent` | above + `POST /agents/<name>` → 422 (route exists, validation fires) |

8. Shutdown app, teardown containers

### CI Configuration

```yaml
# ci.yml — runs on every PR
jobs:
  scaffold-smoke:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        preset: [minimal, standard, full, agent]
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v4
      - run: uv pip install -e .
      - run: python -m pytest tests/integration/test_scaffold_smoke.py -k ${{ matrix.preset }}

# e2e.yml — release gate
jobs:
  e2e:
    runs-on: ubuntu-latest
    services: []  # testcontainers manages its own containers
    strategy:
      matrix:
        preset: [minimal, standard, full, agent]
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v4
      - run: uv pip install -e ".[db-postgres,auth-jwt]" testcontainers
      - run: python -m pytest tests/integration/test_e2e.py -k ${{ matrix.preset }}
```

### Test File Layout

```
tests/integration/
    test_scaffold_smoke.py    — Tier 1: compile + import checks
    test_e2e.py               — Tier 2: testcontainers + endpoint hits
    conftest.py               — shared fixtures (scaffold_project, pg_container, redis_container)
```

### Dependencies

- Tier 1: no new deps (uses stdlib `py_compile`, `tomllib`, `subprocess`)
- Tier 2: `testcontainers[postgres,redis]` in dev dependency group

```toml
[dependency-groups]
dev = [
    # ... existing ...
    "testcontainers[postgres,redis]>=4.0",
]
```

## Consequences

- Template regressions caught early (Tier 1, every PR, no Docker needed)
- Integration bugs caught before release (Tier 2, Docker required)
- Tier 2 is NOT a hard block on PRs — only on releases. Developers without Docker can still contribute.
- New presets or copier questions must add corresponding assertions to both tiers
- Testcontainers requires Docker socket access — CI runners must have it (GitHub Actions `ubuntu-latest` does by default)

## Rejected Alternatives

- **Only Tier 1** — rejected: can't catch migration failures, Redis connectivity issues, or auth flow bugs
- **Only Tier 2** — rejected: too slow and heavy for every PR; Docker requirement blocks contributors without it
- **docker-compose based tests** — rejected: testcontainers is simpler (no compose files to maintain, automatic port allocation, cleanup)
- **Test against real cloud services** — rejected: flaky, requires credentials in CI, expensive
