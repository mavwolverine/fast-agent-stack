# ADR-039 — Embedding-Local Backend: fastembed

**Status:** Accepted  
**Phase:** 5  
**Resolves:** ND-2 (embedding-local backend has no named dependency)

## Context

ARCHITECTURE.md declares an `embedding-local` extras group with the note "no extra deps — uses local model." This is incorrect: there is no standard-library way to run a neural embedding model. A real local embedding backend requires a library that handles model download, tokenization, and inference.

Options considered:

| Library | Weight | GPU required | Runtime | Notes |
|---------|--------|-------------|---------|-------|
| `sentence-transformers` | ~2GB (PyTorch) | No (slow on CPU) | PyTorch | Most popular; very heavy |
| `fastembed` | ~50MB (ONNX Runtime) | No | ONNX | Lightweight, CPU-optimized, maintained by Qdrant |
| `transformers` + `onnxruntime` | ~500MB | No | ONNX | Full HuggingFace stack; high config burden |
| Stub (random vectors) | 0 | No | None | Useless for real retrieval |

## Decision

The `embedding-local` extras group installs `fastembed>=0.8`. The `LocalEmbeddingBackend` uses `fastembed.TextEmbedding` for inference.

```toml
# pyproject.toml
[project.optional-dependencies]
embedding-local = ["fastembed>=0.8"]
```

**Why fastembed:**
1. **Lightweight** — ONNX Runtime, no PyTorch. ~50MB vs ~2GB for sentence-transformers.
2. **CPU-first** — designed for serverless/container environments without GPU. Ideal for development and small-scale production.
3. **Fast** — ONNX Runtime with quantized models; competitive with GPU-backed alternatives on small batches.
4. **Maintained** — actively developed by Qdrant team; Python 3.11–3.14 support confirmed.
5. **Simple API** — `TextEmbedding(model_name).embed(texts)` returns numpy arrays (converted to `list[float]` per ADR-038).

**Settings fields:**

```python
# In BaseSettings
embedding_model: str = "BAAI/bge-small-en-v1.5"  # default model (384 dimensions, ~33MB)
embedding_cache_dir: str | None = None            # model download cache; None = fastembed default
```

**Implementation sketch:**

```python
from fastembed import TextEmbedding

class LocalEmbeddingBackend:
    def __init__(self, settings) -> None:
        try:
            from fastembed import TextEmbedding
        except ImportError:
            raise ImportError(
                "Local embedding backend requires fastembed. "
                "Install it with: pip install fast-agent-stack[embedding-local]"
            )
        self._model = TextEmbedding(
            model_name=settings.embedding_model,
            cache_dir=settings.embedding_cache_dir,
        )
        self._dimensions = self._model.dim

    @property
    def dimensions(self) -> int:
        return self._dimensions

    async def embed(self, text: str) -> list[float]:
        # fastembed is synchronous; wrap in thread executor for I2 compliance
        import asyncio
        loop = asyncio.get_running_loop()
        result = await loop.run_in_executor(
            None, lambda: list(self._model.embed([text]))[0].tolist()
        )
        return result

    async def embed_batch(self, texts: list[str]) -> list[list[float]]:
        import asyncio
        loop = asyncio.get_running_loop()
        results = await loop.run_in_executor(
            None, lambda: [v.tolist() for v in self._model.embed(texts)]
        )
        return results
```

**Async compliance (I2):** fastembed's inference is CPU-bound and synchronous. The backend wraps calls in `asyncio.loop.run_in_executor(None, ...)` to avoid blocking the event loop. This is the same pattern used by the LiteLLM backend's `count_tokens` (Phase 4b).

**Model download:** fastembed downloads ONNX models on first use (~33MB for the default model). This happens during the first `embed()` call, not at construction time. The `embedding_cache_dir` setting allows controlling the download location (useful for Docker builds with pre-cached models).

## Consequences

- ARCHITECTURE.md must be corrected: `embedding-local` extras line changes from `*(no extra deps — uses local model)*` to `fastembed>=0.8`.
- The `embedding-local` extra is no longer "zero deps" — it installs `fastembed` and its transitive deps (primarily `onnxruntime`, `tokenizers`, `numpy`).
- Default model (`BAAI/bge-small-en-v1.5`) is a good balance of quality and size for development/testing. Production users can override via `embedding_model` setting.
- GPU acceleration is available via `fastembed-gpu` (CUDA) but is NOT part of the extras group — users who want GPU install it manually. The backend code is unchanged (fastembed auto-detects providers).
- `run_in_executor` adds thread-pool overhead (~0.1ms per call) but preserves async safety. For high-throughput production, users should prefer the Bedrock or OpenAI embedding backends.
- Rules out `sentence-transformers` (too heavy for a framework default), `transformers` (too much configuration), and stub implementations (useless).

## Rejected Alternatives

- **No dependency / stub** — rejected: a stub returning random vectors is useless for testing RAG pipelines and misleads users into thinking local embedding "works."
- **`sentence-transformers`** — rejected: 2GB+ install (PyTorch), excessive for a framework extra. Users who want it can use ADR-012 dotted-path with their own wrapper.
- **`transformers` + `onnxruntime` manual** — rejected: requires users to handle tokenizer config, model loading, pooling strategy. fastembed encapsulates all of this.
- **Make it optional/undeclared** — rejected: violates the spec's promise of a working `embedding-local` backend. If it's in the extras table, it must work.
