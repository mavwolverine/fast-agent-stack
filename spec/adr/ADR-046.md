# ADR-046 — Agent Tool Calling

**Status:** Accepted  
**Phase:** 10 (tutorial prerequisite)  
**Amends:** Phase 4c agent lifecycle

## Context

The current `@app.agent()` decorator registers handlers that receive messages and return text (non-streaming) or yield text chunks (streaming). There is no mechanism for an agent to invoke tools during generation — the handler is a single function that produces output.

For agentic behavior (Phase 10 tutorial Part 5), the chat agent needs to:
1. Receive a user question
2. Decide it needs to search the document store (tool use)
3. Call the retrieval tool, get results
4. Synthesize an answer from the results
5. Stream the response

This requires a way to define tools that the LLM can call, and a dispatch loop that executes tool calls and feeds results back to the LLM.

## Decision

### Tool Definition

Tools are plain async functions decorated with `@tool`:

```python
from fast_agent_stack.core.ai.tools import tool

@tool(description="Search the document store for relevant passages")
async def search_documents(query: str, top_k: int = 5) -> str:
    """Search indexed documents and return relevant chunks."""
    results = await rag.retrieve(collection="docs", query=query, top_k=top_k)
    return "\n\n".join(r.content for r in results)

@tool(description="Get metadata about an uploaded document")
async def get_document_info(document_id: str) -> str:
    """Return title, page count, and upload date for a document."""
    ...
```

The `@tool` decorator:
- Extracts the function signature (name, parameters, types) for the LLM's tool schema
- Wraps the function for invocation during the agent loop
- Uses the docstring as the tool description if `description` is not provided

### Agent with Tools

```python
@app.agent("chat", backend=_llm, tools=[search_documents, get_document_info])
async def chat(messages: list[Message], *, user_id, api_key_id, conversation_id):
    async for chunk in agent_loop(_llm, messages, tools=[search_documents, get_document_info]):
        yield chunk
```

### Agent Loop

`agent_loop` is the core dispatch function that handles the LLM-tool interaction cycle:

```python
from fast_agent_stack.core.ai.tools import agent_loop

async def agent_loop(
    backend: LLMBackend,
    messages: list[Message],
    *,
    tools: list[Tool],
    max_iterations: int = 10,
) -> AsyncIterator[str | CompletionResult]:
    """Run an LLM-tool loop until the model produces a final text response.

    1. Call LLM with messages + tool schemas
    2. If LLM returns tool calls: execute them, append results to messages, goto 1
    3. If LLM returns text: yield it (streaming) and emit CompletionResult sentinel
    4. Stop after max_iterations to prevent infinite loops
    """
    ...
```

### Tool Schema Generation

The `@tool` decorator generates an OpenAI-compatible function schema from the Python function signature:

```python
{
    "type": "function",
    "function": {
        "name": "search_documents",
        "description": "Search the document store for relevant passages",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "top_k": {"type": "integer", "default": 5}
            },
            "required": ["query"]
        }
    }
}
```

Type mapping: `str` -> `"string"`, `int` -> `"integer"`, `float` -> `"number"`, `bool` -> `"boolean"`, `list` -> `"array"`.

### LLMBackend Amendment

`LLMBackend.complete()` and `LLMBackend.stream()` gain an optional `tools` parameter:

```python
class LLMBackend(Protocol):
    async def complete(
        self, messages: list[Message], *, tools: list[dict] | None = None, **kwargs
    ) -> CompletionResult | ToolCallResult: ...

    async def stream(
        self, messages: list[Message], *, tools: list[dict] | None = None, **kwargs
    ) -> AsyncIterator[str | CompletionResult | ToolCallResult]: ...
```

`ToolCallResult` is a new dataclass:

```python
@dataclass(frozen=True)
class ToolCallResult:
    """Returned by LLM when it wants to call a tool instead of generating text."""
    tool_calls: list[ToolCall]

@dataclass(frozen=True)
class ToolCall:
    id: str
    name: str
    arguments: dict[str, Any]
```

When `tools` is None, behavior is unchanged (backwards compatible).

### Module Location

```
core/ai/tools/
    __init__.py     — tool decorator, Tool class, agent_loop, ToolCall, ToolCallResult
```

### Message Types for Tool Results

Tool results are fed back to the LLM as messages:

```python
Message(role="tool", content=result_text, tool_call_id=call.id)
```

The `Message` dataclass gains optional fields:

```python
@dataclass(frozen=True)
class Message:
    role: str       # "user" | "assistant" | "system" | "tool"
    content: str
    tool_call_id: str | None = None      # for role="tool" responses
    tool_calls: list[ToolCall] | None = None  # for role="assistant" with tool use
```

## Consequences

- `@tool` decorator and `agent_loop` exported from `fast_agent_stack.core.ai.tools`
- `LLMBackend` protocol gains optional `tools` kwarg (backwards compatible)
- `Message` dataclass gains optional `tool_call_id` and `tool_calls` fields (backwards compatible)
- `ToolCallResult` and `ToolCall` exported from `fast_agent_stack.core.ai.llm`
- All four LLM backends (Bedrock, OpenAI, Anthropic, LiteLLM) must handle `tools` parameter and return `ToolCallResult` when the model invokes a tool
- `agent_loop` handles the iteration cycle; individual handlers don't need to implement it manually
- `max_iterations` prevents infinite tool-call loops (safety guardrail)
- Tutorial Part 5 uses this to build an agent that searches documents before answering

## Rejected Alternatives

- **External orchestration (LangChain, Microsoft Agent Framework)** — rejected: adds heavy dependencies for what is a simple loop. The framework should provide its own lightweight tool-call loop.
- **Tools as separate agents** — rejected: over-engineering. Tools are functions, not agents. An agent uses tools; a tool doesn't need its own lifecycle.
- **Synchronous tool execution only** — rejected: tools may call async services (vector store, APIs). All tools are async.
- **No `agent_loop` helper (let users write their own)** — rejected: the loop is boilerplate that every agentic handler needs. Providing it reduces errors and teaches the pattern.
