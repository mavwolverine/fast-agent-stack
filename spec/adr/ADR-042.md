# ADR-042 — UsageService.get_usage() Query API

**Status:** Accepted  
**Phase:** 6  
**Resolves:** Undefined `get_usage` signature (referenced in ADR-035, ROADMAP, GLOSSARY)

## Context

ADR-035 states: "Framework provides a `UsageService` with `log_usage()` (Phase 4c) and `get_usage(user_id, period)` (Phase 6) methods." The `log_usage()` method is implemented. `get_usage()` has no defined signature, return type, or aggregation semantics.

Use cases for `get_usage`:
1. Dashboard display — "you've used X tokens today/this month"
2. Budget enforcement — "reject if user exceeds N tokens per day"
3. Per-agent breakdown — "which agent is most expensive?"
4. Billing — "total cost for user in period"

## Decision

### Return Type

```python
from dataclasses import dataclass
from datetime import datetime

@dataclass(frozen=True)
class UsageSummary:
    """Aggregated token usage for a query period."""
    total_tokens: int
    prompt_tokens: int
    completion_tokens: int
    total_cost_microcents: int   # sum of cost_microcents (nullable rows treated as 0)
    request_count: int           # number of LLM calls in the period
    period_start: datetime
    period_end: datetime


@dataclass(frozen=True)
class UsageByModel:
    """Per-model breakdown within a usage query."""
    model: str
    total_tokens: int
    prompt_tokens: int
    completion_tokens: int
    cost_microcents: int
    request_count: int
```

### Implementation Notes

Both methods execute a single SQL query using SQLAlchemy `func.sum()` / `func.count()` over `token_usage_log` with the provided filters. The `created_at` index makes time-range queries efficient.

```sql
-- get_usage equivalent
SELECT
    COALESCE(SUM(total_tokens), 0),
    COALESCE(SUM(prompt_tokens), 0),
    COALESCE(SUM(completion_tokens), 0),
    COALESCE(SUM(cost_microcents), 0),
    COUNT(*)
FROM token_usage_log
WHERE user_id = :user_id
  AND created_at >= :period_start
  AND created_at <= :period_end;
```

Default period: last 24 hours (`period_start = now - 24h`, `period_end = now`).

### I21 Does NOT Apply to get_usage

I21 (write failures must not abort LLM response) applies only to `log_usage()`. `get_usage()` is a read operation called explicitly by user code (dashboard routes, budget checks) — exceptions should propagate normally.

## Consequences

- `UsageSummary` and `UsageByModel` are exported from `fast_agent_stack.core.ai.usage`
- `get_usage()` requires at least one filter to prevent runaway queries
- No pagination needed — returns a single aggregate row (or one row per model for `get_usage_by_model`)
- Budget enforcement is a user-code concern: call `get_usage()`, compare against threshold, reject if over. The framework provides the query, not the policy.
- Future phases may add `get_usage_by_day()` for time-series data — this ADR doesn't preclude it
- GLOSSARY.md `UsageService` entry already references `get_usage(user_id, period)` — signature is now defined

## Rejected Alternatives

- **Return raw rows (`list[TokenUsageLog]`)** — rejected: forces callers to aggregate; wasteful for common "how much did I use?" queries
- **Single method returning both summary + breakdown** — rejected: separate methods are clearer and allow callers to skip the GROUP BY when they only need totals
- **Streaming/paginated results** — rejected: aggregates are always small (one row for summary, ~5-20 rows for by-model)
- **Automatic budget enforcement in the framework** — rejected: policy decisions (limits, actions on exceed) belong in user code; the framework provides data, not policy
