# ADR-040 — RagService API and Composition Model

**Status:** Accepted  
**Phase:** 5  
**Resolves:** ND-3 (RagService API and composition model unspecified)

## Context

ARCHITECTURE.md Module 11 names the RAG pipeline as a "Composable retrieval service" and places it at `core/ai/rag/`. It lists pluggable vector stores, embedding backends, and document extraction, but defines no API, no protocol, no chunking strategy, and no composition model. Without these, Phase 5 cannot be implemented.

Key design questions resolved here:
1. Is RagService a Protocol (pluggable via ADR-012) or a concrete service?
2. How are embedding, vector store, and extraction wired together?
3. What chunking strategies are supported?
4. What is the public API for ingestion and retrieval?

## Decision

### RagService is a Concrete Service, Not a Protocol

RagService is **not** a pluggable backend family. It is a concrete composition layer that orchestrates the pluggable backends (EmbeddingProtocol, VectorStoreProtocol, StorageProtocol). Users customize RAG by swapping the underlying backends, not by replacing RagService itself.

Rationale: RAG is a pipeline pattern, not an interface pattern. The variability is in the components (which embedding model, which vector store), not in the orchestration logic (embed → store → search → retrieve). A Protocol for the orchestrator would add indirection without value.

### Public API

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class RagChunk:
    """A retrieved chunk with its context."""
    content: str
    score: float
    metadata: dict[str, str | int | float | bool]
    document_id: str | None  # source document identifier
    chunk_index: int          # position within the source document


@dataclass(frozen=True)
class IngestResult:
    """Result of ingesting a document."""
    document_id: str
    chunks_stored: int
    collection: str


class RagService:
    def __init__(
        self,
        embedding: EmbeddingProtocol,
        vector_store: VectorStoreProtocol,
        *,
        chunk_size: int = 512,
        chunk_overlap: int = 64,
        chunking_strategy: str = "fixed",
    ) -> None:
        """
        Args:
            embedding: Embedding backend (any EmbeddingProtocol implementation)
            vector_store: Vector store backend (any VectorStoreProtocol implementation)
            chunk_size: Target chunk size in tokens (approximate, via whitespace split)
            chunk_overlap: Overlap between adjacent chunks in tokens
            chunking_strategy: "fixed" (default) or "paragraph"
        """
        ...

    async def ingest(
        self,
        collection: str,
        content: str,
        *,
        document_id: str | None = None,
        metadata: dict[str, str | int | float | bool] | None = None,
    ) -> IngestResult:
        """Chunk text, embed chunks, and upsert into the vector store.

        Args:
            collection: Target collection name (created if it doesn't exist)
            content: Raw text content to ingest
            document_id: Optional ID for the source document; auto-generated (UUID) if None
            metadata: Optional metadata applied to all chunks from this document

        Returns:
            IngestResult with the document_id and number of chunks stored
        """
        ...

    async def ingest_file(
        self,
        collection: str,
        data: bytes,
        *,
        filename: str,
        content_type: str = "application/octet-stream",
        document_id: str | None = None,
        metadata: dict[str, str | int | float | bool] | None = None,
    ) -> IngestResult:
        """Extract text from a file, then ingest.

        Delegates to the appropriate extractor based on content_type:
        - application/pdf → PDF extractor (requires extract-pdf extra)
        - application/vnd.openxmlformats-officedocument.wordprocessingml.document → DOCX
        - application/vnd.openxmlformats-officedocument.spreadsheetml.sheet → XLSX
        - message/rfc822 → EML (stdlib, no extra)
        - text/* → direct text (no extraction needed)

        Raises UnsupportedFileTypeError for unknown content types.
        """
        ...

    async def retrieve(
        self,
        collection: str,
        query: str,
        *,
        top_k: int = 5,
        filter: dict[str, str | int | float | bool] | None = None,
    ) -> list[RagChunk]:
        """Embed the query and search the vector store.

        Args:
            collection: Collection to search
            query: Natural-language query text (will be embedded internally)
            top_k: Maximum number of chunks to return
            filter: Optional metadata filter passed to vector store

        Returns:
            List of RagChunk results ordered by relevance (highest score first)
        """
        ...

    async def delete_document(
        self,
        collection: str,
        document_id: str,
    ) -> int:
        """Delete all chunks belonging to a document.

        Returns the number of chunks deleted.
        """
        ...
```

### Composition Model

RagService takes its dependencies explicitly at construction time (dependency injection):

```python
# In app setup / lifespan
from fast_agent_stack.core.ai.rag import RagService
from fast_agent_stack.core.vector import get_vector_store
from fast_agent_stack.core.ai.embedding import get_embedding_provider

embedding = get_embedding_provider(settings)
vector_store = get_vector_store(settings)

rag = RagService(
    embedding=embedding,
    vector_store=vector_store,
    chunk_size=settings.rag_chunk_size,
    chunk_overlap=settings.rag_chunk_overlap,
    chunking_strategy=settings.rag_chunking_strategy,
)
```

RagService does **not** call factory functions internally. It receives fully constructed backends. This makes testing trivial (pass mocks) and follows the explicit-dependency pattern used throughout the framework.

### Chunking Strategies

Two built-in strategies, selectable via `chunking_strategy` setting:

| Strategy | Behavior |
|----------|----------|
| `"fixed"` (default) | Split text into chunks of ~`chunk_size` tokens with `chunk_overlap` overlap. Split on sentence boundaries when possible; fall back to whitespace. |
| `"paragraph"` | Split on double-newlines (`\n\n`). Merge short paragraphs up to `chunk_size`. Split long paragraphs at sentence boundaries. No overlap (paragraph boundaries are natural break points). |

Token counting is approximate — uses whitespace splitting (word count ÷ 0.75 ≈ token count). Exact tokenization is model-specific and would add a dependency on the embedding model's tokenizer at chunk time, which is unnecessary for chunking quality.

Chunking is implemented in `core/ai/rag/chunking.py` as pure functions:

```python
def chunk_fixed(text: str, chunk_size: int, overlap: int) -> list[str]: ...
def chunk_paragraph(text: str, chunk_size: int) -> list[str]: ...
```

### Document Extraction Integration

RagService delegates file extraction to the extractors in `core/ai/extraction/`:

```python
# core/ai/extraction/__init__.py
class ExtractionProtocol(Protocol):
    async def extract(self, data: bytes) -> str:
        """Extract text content from binary file data."""
        ...
```

Built-in extractors:
- `PdfExtractor` — requires `extract-pdf` extra (`pdfplumber>=0.10`)
- `DocxExtractor` — requires `extract-docx` extra (`python-docx>=1.1`)
- `XlsxExtractor` — requires `extract-xlsx` extra (`openpyxl>=3.1`)
- `EmlExtractor` — stdlib `email` module, no extra required

Each extractor is extras-gated (I3). `ingest_file` resolves the extractor from `content_type` via a registry. Unknown types raise `UnsupportedFileTypeError`.

### Settings Fields

```python
# Added to BaseSettings
rag_chunk_size: int = 512
rag_chunk_overlap: int = 64
rag_chunking_strategy: str = "fixed"  # "fixed" | "paragraph"
```

### Chunk ID Scheme

Each chunk is stored in the vector store with a deterministic ID:

```
{document_id}:{chunk_index}
```

This enables:
- `delete_document` to enumerate and delete all chunks for a document
- Idempotent re-ingestion (same document_id re-upserts, replacing old chunks)
- Chunk-to-document traceability in search results

`document_id` defaults to `uuid4().hex` if not provided by the caller.

### Error Types

Defined in `core/ai/rag/__init__.py`:
- `UnsupportedFileTypeError(Exception)` — raised by `ingest_file` for unknown content types
- `CollectionNotFoundError` — re-exported from `core/vector/` (ADR-038)

### Not a Protocol (ADR-012 dotted-path not applicable)

RagService is not registered via the dotted-path factory pattern. Users who need a fundamentally different RAG pipeline (e.g., graph-based retrieval, hypothetical document embeddings) compose their own service from the same building blocks (EmbeddingProtocol + VectorStoreProtocol). The framework provides the components; users own the orchestration if the default doesn't fit.

## Consequences

- `core/ai/rag/` contains: `__init__.py` (RagService, RagChunk, IngestResult), `chunking.py` (strategies)
- `core/ai/extraction/` contains: `__init__.py` (ExtractionProtocol, registry), `pdf.py`, `docx.py`, `xlsx.py`, `eml.py`
- RagService is importable from `fast_agent_stack.core.ai.rag`
- No new extras group needed — RagService itself has no dependencies beyond the backends it composes. The backends are independently gated.
- Agent handlers can use RagService for retrieval-augmented generation by calling `rag.retrieve()` to get context, then passing it to the LLM.
- ARCHITECTURE.md Module 11 must be updated with the API documented here.
- Chunking strategies are extensible (users can subclass or replace the chunking functions) but the framework ships only `fixed` and `paragraph`.
- `ingest_file` requires the relevant extraction extra to be installed; otherwise it raises `ImportError` with the install instruction (I3).

## Rejected Alternatives

- **RagService as a Protocol (ADR-012 pluggable)** — rejected: the variability is in the components, not the orchestrator. Making it pluggable adds indirection without solving a real user need. Users who need different RAG patterns compose directly from EmbeddingProtocol + VectorStoreProtocol.
- **LangChain-style chains** — rejected: heavy abstraction, opaque execution, large dependency tree. Our approach is explicit function calls with clear data flow.
- **Semantic chunking (embedding-based)** — deferred: requires running the embedding model during chunking, which is expensive and couples chunking to the embedding backend. May be added as a third strategy in a future phase.
- **Streaming retrieval** — deferred: `retrieve()` returns a list. For the expected result sizes (5–20 chunks), streaming adds complexity without benefit.
- **Built-in reranking step** — deferred: reranking is valuable but requires a cross-encoder model or API call. Can be added as an optional post-retrieval step in a future phase without breaking the current API.
- **Store original files in StorageProtocol** — deferred: RagService currently operates on extracted text only. A future `ingest_and_store` method could persist the original file via StorageProtocol for download, but this couples storage lifecycle to RAG lifecycle in a way that may not fit all use cases.
