# ADR-048: User Migration Branch with Scaffold-Time Head Resolution

**Status:** Accepted
**Date:** 2026-07-13
**Deciders:** Author
**Amends:** ADR-044 (user branch definition)

## Context

ADR-044 introduced branched Alembic migrations for framework tables, with user migrations on the
"main branch" (unnamed). This works for `fas migrate` (`upgrade "heads"`) but breaks
`fas makemigrations`: Alembic's `revision` command calls `get_current_head()` (singular) and fails
with "Multiple heads are present" when framework branches exist.

Additionally, multiple user projects sharing a database need distinct branch labels to avoid
migration ID collisions.

## Decision

This ADR **amends ADR-044's user branch definition**: user migrations move from the unnamed main
branch to a named `{project_name}` branch established by a scaffold-time seed migration.

### 1. Seed migration generated at scaffold time

`fas new` generates an empty seed migration that establishes the user's branch:

```python
# alembic/versions/0001_{project_name}_initial.py
"""Establish {project_name} migration branch.

This is an ordering constraint, not a completeness guarantee. If the framework
later ships new migrations (e.g., fas_auth_0002), they will be applied by
`fas migrate` (which runs "heads") regardless of this depends_on list.
depends_on pins to the framework heads that existed when this project was
scaffolded.
"""
revision: str = "{project_name}_0001"
down_revision = None
branch_labels = ("{project_name}",)
depends_on = ({filtered_framework_heads})


def upgrade() -> None:
    pass


def downgrade() -> None:
    pass
```

The seed migration is always a no-op. User schema changes go in subsequent migrations generated
by `fas makemigrations`.

### 2. Framework heads resolved and filtered at scaffold time

`cli/new.py` resolves framework heads after gathering copier data, filtering by the user's
feature selections:

```python
heads = []
if data.get("include_auth"):
    heads.append("fas_auth_0001")  # resolved from auth branch
if data.get("llm_provider") != "none" or data.get("vector_db") != "none":
    heads.append("fas_ai_0002")    # resolved from ai branch
```

The filtering rule: a framework branch head is included in `depends_on` **only if the
corresponding feature is enabled at scaffold time**. This matches the gate-on-importability
rule from ADR-044: if auth isn't installed, auth migrations aren't in `version_locations`,
so referencing `fas_auth_0001` in `depends_on` would cause a `ResolutionError`.

If no framework features are enabled (e.g., `minimal` preset with no auth, no AI), `depends_on`
is `None` and the user branch is fully independent.

### 3. `makemigrations` targets the user branch

`fas makemigrations` reads the project name from `.copier-answers.yml` and uses the branch label:

```python
command.revision(
    cfg,
    autogenerate=True,
    message=message,
    version_path=user_versions,
    head=f"{project_name}@head",
)
```

No special-casing for first vs subsequent migrations.

### 4. Project name discovery

`makemigrations` reads the project name from `.copier-answers.yml` (always present in scaffolded
projects):

```python
import yaml
answers = yaml.safe_load(Path(".copier-answers.yml").read_text())
project_name = answers["project_name"]
```

## Supersession

This ADR amends ADR-044 section "One Alembic instance manages" line 21:

- **Before:** "User migrations — main branch, generated by `fas makemigrations`"
- **After:** "User migrations — `{project_name}` branch (ADR-048), generated by `fas makemigrations`"

ADR-044's architecture (branched framework migrations, `version_locations` discovery,
`upgrade "heads"`, `include_object` filtering) remains unchanged. Only the user branch
definition changes.

## Rejected Alternatives

**Merge migration:** Generate a migration that merges all framework heads into one, then user
chains from that. Rejected because it creates a file in the framework's version directories or
pollutes the user's directory with framework-coupling logic.

**`head="base"` without branch label:** Creates an unnamed independent branch. Works for a single
project but collides if two projects share a database. Also makes `makemigrations` brittle since
subsequent calls still see multiple heads.

**Hardcoded depends_on in template:** Embedding `fas_auth_0001` directly in the `.jinja` file.
Breaks if framework revisions are renumbered in a future release. Scaffold-time resolution is
strictly more correct.

## Consequences

### Files affected

| File | Change |
|------|--------|
| `cli/new.py` | Resolve filtered framework heads, pass to copier data |
| `cli/db.py` | `makemigrations` reads project name, uses `{project_name}@head` |
| Template `alembic/versions/0001_{{project_name}}_initial.py.jinja` | New seed migration template |
| `spec/INVARIANTS.md` I16 | Amend: user branch is now `{project_name}`, not "main branch" |
| `spec/DX.md` | Update generated project tree to show seed migration |
| `spec/SCENARIOS.md` | Add scenario for seed migration + makemigrations flow |
| `spec/GLOSSARY.md` | Add "seed migration" and "user migration branch" terms |

### Behavioral consequences

- Every scaffolded project gets a unique Alembic branch label (the project name)
- `fas makemigrations` never encounters "Multiple heads" errors
- Multiple projects can share a database without migration ID collisions
- `depends_on` is conditional on scaffold-time feature selections — no ResolutionError
- The seed migration is always a no-op; `fas makemigrations` generates the first real schema
- Adding new framework migrations in future versions requires no changes to existing projects
- Template has no hardcoded revision IDs
